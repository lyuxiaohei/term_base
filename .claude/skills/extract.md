# 术语提取技能

## 触发条件
用户说 "提取术语"、"解析术语"、"/extract" 时执行。

## 工作流程

### 第一步：扫描素材
1. 列出 `src/pending/` 目录下所有文件（不含 processed/ 子目录）
2. 支持的格式：`.txt` `.md` `.docx` `.xlsx` `.xls` `.pdf` `.mp3` `.wav` `.m4a` `.flac` `.ogg`
3. 如果没有待处理文件，告知用户并停止
4. 向用户展示待处理文件列表，确认后继续

### 第二步：读取文件内容
- `.txt` `.md` — 直接读取
- `.docx` — 用 python-docx 读取段落文本
- `.xlsx` `.xls` — 用 openpyxl 遍历工作表
- `.pdf` — 用 PyPDF2 或 pdfplumber 提取文本
- 音频文件 — 尝试用 openai-whisper 或 ffmpeg + whisper 转录；如环境不支持则提示用户先手动转录为 txt
- 遇到读取失败的文件，跳过并记录，继续处理其他文件

### 第三步：AI 提取术语
对所有文件内容进行 AI 分析，提取业务术语。每个术语需包含：
- **术语名称**：简洁准确的业务术语名称
- **所属系统**：从以下类别中选择最匹配的：供应链平台、电商平台、财务系统、人力资源、生产制造、其他
- **业务定义**：清晰完整的术语定义，包含计算方式或业务说明
- **关联术语**：如果内容中提到了其他术语，用 Obsidian 双链 `[[术语名]]` 标注关系

### 第四步：去重检测
1. 扫描 `terms/` 下已有的 `.md` 文件（文件名即术语名）
2. 将提取结果与已有术语对比
3. 对每个术语标注状态：
   - **新增** — 术语库中不存在
   - **重复** — 已有同名术语（显示现有定义供对比）
4. 向用户展示去重结果

### 第五步：写入文件
对用户确认导入的术语，每个术语生成一个 Markdown 文件：

**文件路径**：`terms/<术语名称>.md`

**文件格式**（严格遵守）：
```
---
system: "<所属系统>"
source: "<来源文件名>"
created: "<今天日期 YYYY-MM-DD>"
updated: "<今天日期 YYYY-MM-DD>"
---

<业务定义>

## 关联
- <关系类型>：[[<关联术语1>]] [[<关联术语2>]]
```

**写入规则**：
- 文件名使用术语名称，如果术语名包含 `/` 替换为 `-`
- 如果文件已存在（重复术语），更新其定义和 updated 字段，保留原有 created 字段
- 关联部分的格式：`- 上游：` `- 下游：` `- 相关：` 根据实际关系填写，没有关联则保留空行

### 第六步：归档源文件
将已处理的源文件移动到 `src/pending/processed/` 目录。

### 第七步：汇报结果
告知用户：
- 处理了几个文件
- 新增了几条术语
- 更新了几条术语
- 跳过了几条（用户未勾选的）
- 提醒用户在 Obsidian 中查看 Graph View 查看术语关系

## 注意事项
- 不要修改 `vault/templates/` 和 `vault/views/` 下的文件
- YAML frontmatter 中的值不要加引号包裹的额外空格
- 关联术语的双链格式必须是 `[[术语名]]`，不要用其他格式
- 如果内容中没有明确的术语关系，不要编造关联
